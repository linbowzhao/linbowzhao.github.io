WebRTC(浏览器点对点视频)简介

发表于 2019-07-08

WebRTC，名称源自网页即时通信（英语：Web Real-Time Communication）的缩写，是一个支持网页浏览器进行实时语音对话或视频对话的API。它于2011年6月1日开源并在Google、Mozilla、Opera支持下被纳入万维网联盟的W3C推荐标准。

https://zh.wikipedia.org/wiki/WebRTC

一、代码解析

总之，WebRTC对于程序员来说就是浏览器的api接口给我们调用，可以让我们在没有中间服务器的情况下进行点对点视频连接，我大致介绍一下这些api接口和相关的一些信息。
先来介绍一下offer.html里的代码

这两个都是一样的api，只是浏览器推荐使用第一个接口，第二个只是为了兼容老版本，已经逐渐在废弃。

新建一个一个由本地计算机到远端的WebRTC连接。该接口提供了创建，保持，监控，关闭连接的方法的实现。

二、关于WebRTC的一些原理和术语

1、WebRTC使用的协议

该图引自《web性能权威指南》，如有侵权，立马删掉。

本文的例子其实是模仿的左边的TCP连接上的通信方式，只是为了方便把WebSocket信令服务器换成了浏览器间的标签通信，TCP当传输层协议比较可靠，但是容易造成延迟。
为了而视频直播，一般要考虑速度问题，业务中一般会选择UDP通信方式，UDP则正好相反，它只负责有什么消息我就传过去，不负责安全，不负责有没有到达，不负责交付顺序，所以速度比较快，这里从底层来看是满足WebRTC的需求的。

2、信令服务器

在TCP的方式中，信令服务器指WebSocket，在UDP方式中，指DataChannel。
这不是用来做视频数据流的中间桥接的，只是用来做给两个通信的浏览器之间做信息桥接的，用来传递用户数据和浏览器之间的ip信息，用来建立连接。

3、ice,stun和turn服务器

stun和turn服务器是用来做内网穿透的，要理解这个内外穿透，我们先要了解NAT网关。
由于ipv4地址的稀缺和互联网用户的激增，为了解决这个问题，诞生了NAT网关，大概就是一个网关是一个ipv4地址，比如你们公司的网关，然后公司内部的设备分配的是内网的地址，这样解决ipv4地址不够用的情况。但是严重削弱了ip端对端的通信能力，内网里的设备只能通过网关访问外网。
也就没法实现浏览器之间端对端通信了，这时候就需要ice,stun和turn服务器来解决这个问题了。这是三种不同的方式吧，具体怎么操作我就不太清楚了。

4、WebRTC泄露真实IP

由于WebRTC的端对端通信，其会泄露真实IP，ice,stun和turn服务器的存在使得防火墙和代理也失去作用。

5、DataChannel

除了传输音频和视频数据，WebRTC 还支持通过 DataChannel API 在端到端之间传 输任意应用数据。DataChannel 依赖于 SCTP(Stream Control Transmission Protocol，流控制传输协议)，而 SCTP 在两端之间建立的 DTLS 信道之上运行的。
DataChannel api调用和WebSocket类似。

web开发安全

发表于 2019-06-27

一、CSRF攻击(跨站请求伪造)

1、CSRF原理

跨站请求伪造（英语：Cross-site request forgery），也被称为 one-click attack 或者 session riding，通常缩写为 CSRF 或者 XSRF， 是一种挟制用户在当前已登录的Web应用程序上执行非本意的操作的攻击方法。
跟跨网站脚本（XSS）相比，XSS 利用的是用户对指定网站的信任，是在目标网站植入黑客的代码获取用户的重要信息或者执行恶意操作。
CSRF 利用的是网站对用户网页浏览器的信任，是利用用户的cookie等登录信息伪造合理的恶意请求来达到自己的目的，黑客中间无法得到用户信息，但是可以完成一次黑客想要达成的请求来获取利益（比如转账请求）。
[1] 用户在目标网站上登录过，目标网站存在CSRF漏洞
[2] 用户点击黑客构造过攻击链接或者页面跳转按钮，发送黑客构造的请求给目标服务器。

2、CSRF漏洞特征

完全靠cookie识别用户，请求参数中没有身份识别的参数。

3、CSRF攻击类型

[1] GET型漏洞
GET类型漏洞是比较危险的，黑客只需要构造攻击链接即可，而且链接域名都是正确的目标网站，用户根据连接无法识别该攻击。
比如这个例子 http://zhifubao.com/pay?toAccount=lin&amount=10000
要是一个支付GET请求只需要转账用户的账号toAccount和金额amount，然后登录信息都是在请求的cookie里面取，那黑客就会构造上面这样的链接，用户点击了之后，会直接打开浏览器发出请求，用户要是之前在这个网站登录过，请求就会带上用户的cookie信息转账给黑客的目标账户了。
不过一般支付的接口都会比较小心，加的验证比较多，这种漏洞也会发生在其它地方，比如投票接口，点赞接口，收藏接口或者一些操作数据库的接口（被用来攻击用户恶意删除用户数据）。
GET类型的CSRF漏洞攻击成本非常低，应当注意。
[2] POST型漏洞
POST型漏洞是利用浏览器的表单提交完成的，表单提交其实是可以跳转到不同域名下的，相当于一次“POST类型的页面跳转”，浏览器的机制是跳转到新的页面，并立即执行一次POST请求。这也是有时候我们刷新页面会提示已经提交过的表单会重新提交的原因，即使url上没有任何参数，这在url上是看不出来区别的，打开http的请求头，我们才可以看到这和直接在浏览器输入请求有什么区别。
POST型CSRF攻击成本比GET型CSRF攻击成本稍高，因为黑客伪造的链接不是在同一个域名下的，是在黑客自己的域名下做的一次跳转，黑客无法直接通过链接发起目标请求，比如下面这个链接。
http://local.gaeamobile.net:8001/linbowzhao.github.io/source/_drafts/web开发安全/CSRF.html

3、CSRF攻击的防御

[1] Token机制
既然黑客是利用浏览器的机制来自动带上用户的cookie信息，那我们不单独依赖cookie来识别用户信息即可，可以把用户表示通过参数传递，黑客没法获取到用户的cookie信息，就没法构造出有效的CSRF链接，但是cookie可被js读取的话又容易受到之前讲过的XSS攻击，个人觉得生成一套两个匹配的的标识，一个叫做token1吧保存在http only的cookie，一个token2保存在js可读取位置，所有的请求带上token2，服务端接收到请求之后比对token1和token2是否匹配，匹配接受请求。
这样的话既防御了XSS攻击获取cookie，又防御了CSRF攻击，个人觉得是比较好的解决办法。
[1] 验证HTTP Referer字段
它记录了该HTTP请求所在的地址，表示HTTP请求从那个页面发出的，验证该请求是从哪个页面发出的，黑客构造的链接会是从黑客的域名而不是目标网站的域名。
这个方法个人觉得不太好，稳定性不高
首先是GET请求的CSRF攻击链接，打个比方，如果是从qq发送的过来的链接或者用户复制粘贴打开的，这里的Referer字段是空的，我们没法验证从哪里跳转过来
第二是一些低级的浏览器比如 IE6可以通过某种方式篡改Referer的值，这就给了攻击者可乘之机
第三是有些用户认为这样会侵犯到他们自己的隐私权，特别是有些组织担心 Referer 值会把组织内网中的某些信息泄露到外网中。因此，用户自己可以设置浏览器使其在发送请求时不再提供 Referer
总之个人觉得Referer限制是个不太稳定的方法，容易误伤正常用户或者有漏网之鱼。
[3] 验证码验证
优点是安全性高，缺点是需要用户操作，不可能每个接口都添加验证码验证，用户体验不好。

二、JSONP劫持

JSONP的原理在这里就不介绍了，需要了解的自行百度。

1、JSONP劫持原理

当这个JSONP接口没有验证Referer头的时候,就存在JSONP劫持漏洞,即在任何域下都能窃取到传输的数据，黑客直接调用JSONP接口进行使用，获取用户信息和执行恶意操作。
JSONP其实也属于CSRF攻击的一种吧，都是被黑客用来伪造恶意请求，从而达到攻击目的。

2、JSONP劫持例子

http://local.gaeamobile.net:8001/linbowzhao.github.io/source/_drafts/web开发安全/JSONP.html

3、JSONP劫持的防御

因为JSONP属于CSRF攻击范围内，所以CSRF攻击的防御方式对JSONP劫持都适用，简单的方法是验证HTTP Referer字段，比较安全的方法是添加Token机制

4、JSONP callback可定义导致的安全问题

callback可自定义的时候如果没有对定义的函数名进行过滤，还会导致xss攻击安全问题，例如下面这个链接
http://local-api.gaeamobile.net:8888/user?callback=< script>alert(‘xss’)</ script>
点开这个链接的时候，用户数据就被窃取去了，所以为了安全问题，还是用固定的callback函数。

三、高级指纹

1、canvas: canvas指纹
利用不同设备不同浏览器在绘制同一张图像时候和差异化信息提取特征码。

从根本上来说，每一种浏览器都会使用不同的图像处理引擎，不同的导出选项，不同的压缩等级，所以每一台电脑绘制出的图形都会有些许不同，这些图案可以被用来给用户设备分配特定编号（指纹），也就是说可以用来识别不同用户。

不同的电脑，显卡，设备分辨率不一样，渲染出来的图片有差异化不足为奇。不同浏览器不同图像的处理引擎，渲染出来的图片不一样也是一样道理。需要注意的是，即使是同一型号的设备，同一个版本的浏览器，渲染出来的图片依旧是有差异化的，肉眼无法辨别罢了，网上数据显示，差异化达到99.4%。个人推测是显卡的特征信息被加入到了图片渲染的过程中，导致了图片细微的差异化信息。

先用canvas生成一张多个半透明的不同颜色的圆交叉的图片，并在上面填充文本，转化成base64，作为生成指纹的种子。

2、webgl:webgl指纹
原理和canvas类似，不同之处是渲染的图片不同，并且调用webgl的api渲染而成。渲染图片为渐变色3D立体图像，为了调用尽可能多的webgl api突出差异化，渲染出来的图片转化为base64，作为生成指纹的种子。

3、audio:audio指纹
利用浏览器AudioContext的api OscillatorNode产生一个三角波，然后将波传Dynamics Compressor Node，一个调节声音的信号处理模块，产生压缩效果。 然后，经处理的音频信号通过Analyser Node转换为频率域。频域在不同的浏览器中是不同的，这个特征受浏览器的影响，不能完全反应出声卡的特征。频率和峰值的比，在浏览器之间是相对稳定的。因此，在频率和值的坐标系上创建一个间距很小的列表，并将峰值频率和峰值映射到相应的格子。 如果一个格子包含一个频率或值，我们将格子标记为1，否则为0，这样的格子列表用作跨浏览器特征。

先用AudioContext.createOscillator 产生一个三角波，然后用 AudioContext.createDynamicsCompressor，调节声音的信号，然后把信号用renderedBuffer.getChannelData 转化为与通道关联的PCM数据，采集4500到5000段稳定的数据作为生成指纹的种子

四、指纹生成

得到以上生成指纹的种子后，全部拼接到一块，用Murmur哈希函数生成32位长度的指纹，不选用md5加密是因为这里不需要md5那样强度的非逆转加密，可以节省生成时间。

3、Undefined类型和Null类型的都只有一个值，即undefined和null；Boolean类型有两个值：true和false；Number类型的值有很多很多；String类型的值理论上有无数个。
4、所有对象都有valueOf()和toString()方法，它们继承自Object，当然也可能被子类重写。

一. 有和无

在图1中，JavaScript值的六种类型用蓝底色的矩形表示。它们首先被分成了两组：
String、Number、Boolean和Object (对应左侧的大矩形框)

左侧是一个存在的世界，右侧是一个空的世界。
所以，左右两个世界中的任意值做==比较的结果都是false是很合理的。(见图1中连接两个矩形的水平线上标的false)

五. 单纯与复杂

原始类型是一种单纯的类型，它们直接了当、容易理解。然而缺点是表达能力有限，难以扩展，所以就有了对象。对象是属性的集合，而属性本身又可以是对象。所以对象可以被构造得任意复杂，足以表示各种各样的事物。
但是，有时候事情复杂了也不是好事。比如一篇冗长的论文，并不是每个人都有时间、有耐心或有必要从头到尾读一遍，通常只了解其中心思想就够了。于是论文就有了关键字、概述。JavaScript中的对象也一样，我们需要有一种手段了解它的主要特征，于是对象就有了toString()和valueOf()方法。
toString()方法用来得到对象的一段文字描述；而valueOf()方法用来得到对象的特征值。
当然，这只是我自己的理解。顾名思义，toString()方法倾向于返回一个字符串。那么valueOf()方法呢？根据规范中的描述，它倾向于返回一个数字——尽管内置类型中，valueOf()方法返回数字的只有Number和Date。

如何优雅地封装js

发表于 2019-05-28

一、es5最有效且较优雅的前端封装方法（构造方法：继承实例属性/方法+原形模式：继承静态属性/方法）

二、各种属性/方法的概念
1、私有属性/方法：只能在定义的类内调用的方法和属性，不能在定义类外被直接调用。
2、公有属性/方法：能被定义类外的方法直接调用，和私有方法相对。
3、静态属性/方法：和私有公有没有直接关系，定义在构造函数的原型上不用实例化即可直接调用的属性/方法。
4、实例属性/方法：和私有公有没有直接关系，和静态方法相对，定义在构造函数上必须被实例化才能调用的函数。

三、各种属性/方法的应用场景
1、私有属性/方法：内部调用的属性/方法，不希望或者不需要泄露出去，只需要在内部调用。
2、公有属性/方法： 内部调用的属性/方法，有静态和实例两种
3、静态属性/方法： 每个实例调用的属性和方法一样，而且指向同一个属性/方法的地址，改变构造方法的属性/方法后，所有实例的属性/方法随之改变。需要维护数据方法统一的情况下使用。
4、实例属性/方法：每个实例调用的方法相似却又有区别，需要实例化传入参数才能调用，如果每个实例是一模一样的方法，就没有必要用实例方法，用静态方法即可。

四、方法优先级
外部调用的时候，如果类里存在同名实例属性/方法和静态属性/方法的时候，实例属性/方法优先于静态属性/方法

五、继承
上面我们讲的例子是原形关系，有时候，项目比较庞大的时候，我们可能会用到继承来拓展方法。
关于继承的方式，这里有篇文章写的很好，我就不再赘述了，这几种方法中，个人觉得call继承和经典继承（道格拉斯继承）结合比较优雅地解决了问题。call继承可以继承父类的实例方法，经典继承可以继承父类的静态方法。
